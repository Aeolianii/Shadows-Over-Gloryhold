# Decisions

## 2026-05-28 进度门槛决策

- 数值进度必须对应明确玩法后果：`investigation_progress` 达到 5/5 时代表第一章核心命案链条已经足够推进，不能只停留为 UI 计数。
- AI 可以补充叙事，但不能阻塞确定性章节门槛；当后端世界状态满足推进条件时，`StoryDirector` 必须能越过缺失的 `advance_chapter` 字段继续推进。
- 前端状态条要表达“已完成/可推进”的含义，而不是继续显示普通进度，降低玩家对下一步的困惑。

## 2026-05-28 结局门槛决策

- 结局判定不能依赖固定 3 票，因为项目常用于单人或少人数测试；实际门槛跟随已选角色玩家数，但最多仍为 3 票。
- 终局 flag 和投票都属于确定性后端门槛，AI 可以写结局文案，但不能让已满足条件的房间停在无推荐行动、无推进反馈的状态。
- 所有主要阵营都应有可被确定性选中的胜利结局；秘术盟对应伊莱亚斯结局 `ending_arcane`。

## 2026-05-28 WebSocket 状态同步决策

- 章节推进消息既是日志事件，也是状态更新；前端处理 `chapter_changed` 时必须同步房间章节状态，不能只把它塞进 `public_log`。
- 后端广播房间状态时应同时提供 `current_chapter_id` 和完整 `current_chapter`，避免前端只能显示标题却无法更新依赖章节 id 的逻辑。

## 2026-05-28 GM 边界决策

- AI GM 要像真实桌面 GM：鼓励合理创意，但不能把玩家愿望直接改写成世界事实。
- 对跳关、直接胜利、索要所有秘密、无条件拿关键物品等要求，后端必须确定性拒绝，并把玩家引导回当前章节可执行行动。
- 章节推进和结局判定必须由世界状态、规则结果或明确 flag 支持，不能只依赖模型返回的 `advance_chapter` 或 `ending_title`。

## 2026-05-28 调查进度决策

- 调查进度是剧情理解进展，不是点击次数；不能由 `RuleEngine` 对每个调查动作机械累加。
- `world_state` LLM route 负责按具体剧情、证据新鲜度和章节目标审核进度变化，普通行动多数只给 0 或 1 点。
- AI 审核失败时也要保守，不允许推荐行动点击流把第一章快速刷满。

## 2026-05-28 UI 决策补充

- GamePage Step 4 继续采用“视觉替换、逻辑保留”：不改 `App.tsx` 的 WebSocket hook，不改后端消息类型，只在 `GamePage` 内把真实日志和发送函数接进新视觉组件。
- 中央“剧情事件台”以 `room.public_log` 为唯一数据源，避免重新维护一份前端 Mock 日志；底部指令台只负责输入状态和临时 pending 文案，实际状态仍由 WebSocket 推送回填。
- 角色头像不再作为小图嵌入，而是作为镜框内全幅裁切图渲染；优先保证头像贴合镜框和人物可读性，背景图只保留环境氛围，不能压在头像视觉层之上。
- 720p 桌面高度下底部卷宗优先展示完整摘要，不显示会被截断的第二条能力。
- Step 3 采用“视觉替换、逻辑保留”的做法：重写 `LobbyPage` / `RolePanel` 的 JSX 和 Tailwind v4 utility 布局，但不改变房间状态、选角状态和开始游戏回调的职责边界。
- 角色长廊不再使用 Mock `CHARACTERS`；所有角色名称、称号、头像、能力和公开身份均来自后端剧本数据。
- 选择角色采用聚焦角色上的悬浮按钮，避免旧滑动逻辑导致头像偏移；左右箭头只改变聚焦角色，不绕过原有选择/锁定流程。
- 底部控制台以真实房间信息和当前角色卷宗为主，房间卡片压缩，给当前角色身份和能力留更多横向空间。

## 产品决策

- 游戏体验优先于网页感；选角页要像黑暗西幻角色大厅。
- 角色是选角页视觉中心，房间信息降级到底部控制台。
- AI 不代答真人已占用角色。
- 玩家秘密身份、目标、线索只给本人。
- 推荐行动是玩家私有辅助，不广播给其他玩家，避免泄露个人目标或初始线索。

## 技术决策

- 前后端分离：React/Vite + FastAPI。
- 房间状态暂用内存，适合 MVP。
- 剧本 JSON 驱动。
- AI 使用 OpenAI 兼容接口，DeepSeek route 分流。
- AI HTTP 默认 `AI_HTTP_TRUST_ENV=false`，避免系统 SOCKS 代理导致失败。
- 普通行动默认关闭 world_state 二次审核：`AI_REFINE_WORLD_STATE=false`。
- Action Parser 对常见关键词优先本地解析。
- NPC 记忆采用摘要 + 最近窗口。
- AI 输出先清洗再进 Pydantic，避免单个坏字段让 WebSocket 崩溃。
- 关键进度类世界状态由后端硬封顶，不能只相信 AI 或前端。
- 推荐行动使用单独 DeepSeek route：`DEEPSEEK_RECOMMENDATION_API_KEY`，失败时走确定性 fallback。

## UI 决策

- 首页灰烬粒子可覆盖 UI，但必须 `pointer-events: none`。
- 选角页固定三段：标题 / 角色大厅 / 底部控制台。
- 修复选角页时优先整齐、固定窗口、不溢出，再做动画细节。
- 推荐行动放在游戏界面左侧玩家列表下方，作为推进辅助而不是主输入替代品。
- 当前美术方向采用 `./前端` 示例：黑金紫、仪式线框、灰烬粒子、展示衬线标题。按用户要求，主界面保留原版视觉，选角页和游戏页保留新版风格。
- 选角页采用固定 7 槽角色镜框，不再保留滑动选择交互；箭头放在标题两侧，头像槽点击和标题箭头共同切换当前聚焦角色。
- Tailwind 使用 v4 Vite 插件方式，不保留旧 PostCSS 接入和 `tailwind.config.js` 内容扫描配置。
- 选角页新增布局优先使用 Tailwind v4 utilities 和 important utilities 覆盖旧 CSS，逐步减少旧 CSS 对关键布局的影响。
- 新视觉整合分步推进：先合并全局 CSS token 和特效，再逐页迁移 Home/Lobby/Game；每一步都保留现有 API、WebSocket、状态管理和生命周期逻辑。
